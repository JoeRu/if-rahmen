# ImmichFrame Build Warnings Fix — Design

## Purpose

`docker compose build` of ImmichFrame's frontend (`immichFrame.Web`) prints
warnings that are reproducible on a clean checkout of upstream
`immichFrame/ImmichFrame:main` (confirmed via `git diff upstream/main` on the
affected files — they are byte-identical, so this is not a fork-specific
issue). The build still succeeds; these are latent-defect warnings, not
build failures. Goal: fix all three findings and contribute the fix back
upstream via a PR.

## Findings

1. **Cosmetic — tsconfig warning.** `vite build` prints `Cannot find base
   config file "./.svelte-kit/tsconfig.json"` on a fresh checkout, because
   `.svelte-kit/` (gitignored, generated by `svelte-kit sync`) doesn't exist
   yet when Vite first reads `tsconfig.json`. Harmless — the final image
   works correctly regardless — but it's missing the standard SvelteKit
   scaffold safeguard for this.

2. **Reactivity warning — `overlay-controls.svelte`.** `shortcutList` (an
   array mapping keyboard keys to the `next`/`back`/`pause`/`showInfo`
   props) is built once as a plain `const` at component init, not
   reactively. The Svelte 5 compiler flags this because destructured props
   are reactive; capturing them in a non-reactive container silently
   freezes them at their initial value. Low real-world impact today (the
   current inline callbacks in `home-page.svelte` behave identically
   however they're invoked), but it's the wrong idiom and the warning
   should be eliminated.

3. **Reactivity bug — `overlay-qr.svelte`.** `imageUrl` (built from the
   `id` prop, used for both the QR code and the "Open in immich" link) is
   computed once at component init, not reactively. `image-overlay.svelte`
   renders `<OverlayQr id={asset.id} />` without a keyed block forcing
   remount on asset change. If the info overlay stays open (or is
   reopened without an intervening full unmount) while the slideshow
   advances underneath, the QR code and link silently keep pointing at
   whatever asset was current the first time this component instance was
   created — a real, user-facing staleness bug, not just noise.

## Fixes

1. **`immichFrame.Web/package.json`** — add `"prepare": "svelte-kit sync"`
   to `scripts`. This is the standard `create-svelte` scaffold entry this
   project is missing. `npm i` triggers the npm `prepare` lifecycle
   automatically, so `.svelte-kit/tsconfig.json` exists before `npm run
   build` (`vite build`) ever reads `tsconfig.json`. No Dockerfile change
   needed — this fixes local `npm run build`/`npm run dev` too.

2. **`overlay-controls.svelte`** — change:
   ```ts
   const shortcutList = [ ... ];
   ```
   to:
   ```ts
   const shortcutList = $derived([ ... ]);
   ```
   so it recomputes whenever `next`/`back`/`pause`/`showInfo` change
   identity, matching the compiler's own suggested fix ("did you mean to
   reference it inside a closure instead?").

3. **`overlay-qr.svelte`** — change:
   ```ts
   let imageUrl = `https://my.immich.app/photos/${id}`;
   ```
   to:
   ```ts
   let imageUrl = $derived(`https://my.immich.app/photos/${id}`);
   ```
   so the QR code and link always reflect the currently displayed asset.

## Workflow

The fork's `origin/main` is currently identical to `upstream/main` (no
sync needed first). Work happens in a new worktree
(`.worktree/fix-build-warnings`, branch `fix-build-warnings`, created off
`main` per this repo's existing worktree convention), keeping the primary
`ImmichFrame/` checkout untouched on `main`. After implementing and
verifying (see below), push the branch to the fork (`origin`) and open a
PR against `immichFrame/ImmichFrame:main`.

## Testing

No frontend test framework exists in this project (no vitest, no
`*.test.*`/`*.spec.*` files) — verification is manual, matching this
project's existing practice:

1. `npm run check` (svelte-check) — confirm it passes, and specifically
   that no `state_referenced_locally` warnings remain for the two changed
   files.
2. `npm run build` — confirm the tsconfig warning is gone and no new
   warnings appear.
3. Run the `local-smoke-test` skill in rebuild-and-test mode against the
   fix branch (temporarily pointing `docker-compose.yml`'s build context
   at the worktree, or building the worktree directly with `docker build`)
   to confirm the app still loads, no console errors, no failed requests,
   and the slideshow still advances.
4. Manual check specific to the `overlay-qr.svelte` fix: open the info
   overlay (press "i" or tap Info), let the slideshow advance to a new
   asset while the overlay is visible (or reopen it after an
   auto-advance), and confirm the QR code and "Open in immich" link now
   point at the *current* asset's id, not the one shown when the overlay
   was first opened.

## Out of Scope

- No change to `Dockerfile` (the `prepare` script fix makes that
  unnecessary).
- No broader refactor of `overlay-controls.svelte`/`overlay-qr.svelte`
  beyond the two `$derived` changes.
- No changes to the fork's `personal-main` branch — this targets `main`/
  upstream directly, since the bug is upstream's.
